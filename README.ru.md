**Как `.gitignore`, только для публичности.**

Нужно полное объяснение всех правил? См. [расширенная настройка](./docs/ADVANCED.md) / [RU](./docs/ADVANCED.ru.md).

У тебя приватный репо. Нужен публичный — без секретов. Эта тулза держит их в
синке. Автоматически.

## Философия проекта

**Чтобы получить пользу, не нужно сначала чему-то учиться.** Установил, запустил
`init`, затем используешь `scan` для проверки и `publish` для публикации.
Безопасное поведение включено по умолчанию: обычный сценарий не требует помнить
кучу параметров, собирать конвейер или разбираться во внутренностях Git.

Когда проект становится сложнее, тот же инструмент остаётся гибким: через
`.gitpublic` можно настроить исключения, замены, allow-list, ветки, теги, hooks,
CI-отчёты и схему remotes, не ломая простой основной сценарий.

На практике это означает:

- легко начать;
- безопасно по умолчанию;
- одна очевидная команда на одно действие;
- полезно без экспертной настройки;
- достаточно универсально для сложных private → public процессов.

## Быстрый старт

```bash
pip install git-private2public
git-private2public init          # создаёт папку .gitpublic/
```

Отредактируй `.gitpublic/config` — source и target. Значения могут быть `owner/repo`, полный Git URL или локальный путь:

```
source = you/private-repo
target = you/public-repo
```

Отредактируй `.gitpublic/ignore` — что прятать, по строке (как `.gitignore`):

```
.env
secrets/
*.key
```

Опубликуй:

```bash
git-private2public publish
```

Готово. Публичный репо чистый.

## Авто-публикация при каждом `git push`

```bash
git-private2public hook enable     # вкл
git push                           # также публикует публичный mirror
git-private2public hook disable    # выкл
```

Нативный git-хук. Без CI, без GitHub Actions. Работает офлайн.

## Guard — предохранитель на pre-push

`guard` ставит лёгкий pre-push хук, который **блокирует** `git push`, если в
сканируемом контенте матчатся известные паттерны секретов. В отличие от
`hook` (который ещё и переписывает историю и force-пушит публичное
зеркало), `guard` — чисто отказ: без clone, без filter-repo, без push.

```bash
git-private2public guard enable    # поставить pre-push хук
git-private2public guard status    # стоит?
git-private2public guard disable   # снять
git-private2public guard run       # ручное сканирование (это же делает хук)
```

**Что сканируется по умолчанию:**
1. Каждый отслеживаемый файл в рабочем дереве (`git ls-files`).
2. **Каждый blob в git-истории** — ловит секреты, закоммиченные в старых
   коммитах, а потом удалённые из HEAD, но не вытертые через filter-repo.
   Чтобы пропустить этот шаг (быстрее, но не ловит старые утечки):
   `git-private2public guard run --no-history`.

**Дефолтные паттерны секретов (всегда включены, `.gitpublic/` не нужен):**
- OpenAI и Anthropic: `sk-...`, `sk-proj-...`, `sk-ant-...`
- GitHub: `ghp_...`, `github_pat_...`, `gho_...`, `ghs_...`, `ghr_...`
- HuggingFace и Slack: `hf_...`, `xox[baprs]-...`
- постоянные и временные ключи AWS: `AKIA...`, `ASIA...`
- Google API/OAuth и GitLab PAT
- токены Telegram Bot, npm, PyPI, Stripe, SendGrid и Discord
- учётные данные Twilio, Mailgun и DigitalOcean
- универсальные JWT и заголовки приватных PEM-ключей

**Свои паттерны** — кладёшь в `.gitpublic/scan`, по строке (литерал или
`regex:...`). Они дополняют дефолтные.

**Allowlist** — всё, что в `.gitpublic/allow`, считается исключением для
широких regex-правил, которые могут зацепить публичный домен.

**Одноразовый bypass (не рекомендуется):**

```bash
GIT_PRIVATE2PUBLIC_SKIP_GUARD=1 git push
```

**Что guard пишет, когда блокирует push:**

Сообщение подстраивается под то, где сидит секрет:

- **В рабочем дереве** — указывает на файл, советует отредактировать
  или добавить в `.gitignore` + `git rm --cached`, напоминает ротировать
  живой секрет **до** коммита.
- **В git-истории** — объясняет, что правки HEAD недостаточно, и печатает
  точную следующую команду:
  - `git-private2public publish` (если `.gitpublic/` настроен), или
  - `git filter-repo --replace-text replacements.txt --force` руками,
    с форматом `replacements.txt`.

Если нарушения и там и там — печатает обе секции.

Если нужен только `guard` (без `hook`/publish) — `guard enable` работает
сам по себе. Они делят один pre-push файл, поэтому `guard enable`
откажется, если уже стоит `hook`; сними один, прежде чем ставить другой.

### Зачем сканировать историю?

Секрет ушёл в коммит месяц назад, потом его убрали из HEAD — но объект
коммита всё ещё лежит в `.git/objects/`. Push HEAD его не отдаст напрямую,
но любой, у кого уже есть репо, увидит его через `git log -p`. История-
скан в guard это ловит. Чтобы **реально стереть** из истории — запусти
`git-private2public publish` (использует filter-repo) или
`git filter-repo --replace-text` руками.

## Папка `.gitpublic/`

Каждый файл — одна забота. Как `.gitignore` — одно правило на строку, `#` для
комментариев. Если файла нет — настройки просто нет.

| Файл | Что внутри | Формат |
|------|------------|--------|
| `config` | source, target, push | `key = value` |
| `ignore` | что НЕ публиковать | путь/маска на строку |
| `replace` | найти → заменить в файлах | `old ==> new` на строку |
| `scan` | отказаться пушить если найдёт | паттерн на строку |
| `allow` | исключения для правил `scan`, которые ловят домены | по одному домену-исключению в строке |

**Простой** — редактируй только `ignore`:

```
.env
secrets/
*.key
```

**Средний** — ещё `replace`:

```
<PRIVATE_IP> ==> 203.0.113.5
real-token ==> ***
regex:[A-Fa-f0-9]{64} ==> ***
```

**Сложный** — ещё `scan` + `allow`:

```
# scan:
regex:github_pat_[A-Za-z0-9_]{30,}
regex:192\.168\.
regex:[a-z0-9.-]+\.[a-z]{2,}

# allow:
github.com
get.docker.com
```

## Команды

```
init        создать .gitpublic/ конфиг
scan        почистить во временный репозиторий, проверить, не пушить
publish     вычистить + запушить
hook        enable / disable / status
guard       enable / disable / status / run  (pre-push сканер секретов)
```

## Как работает `allow` / domains

Ничего не автоблокируется просто потому, что это домен.

`allow` — это список исключений для `scan`. Если `.gitpublic/scan` отсутствует или пустой, `allow` ничего не делает и домены вообще не проверяются.

Чтобы блокировать похожие на домены строки, добавь широкое regex-правило в `.gitpublic/scan`:

```
regex:[a-z0-9.-]+\.[a-z]{2,}
```

Теперь любой найденный домен валит проверку, кроме случаев, когда сам найденный домен перечислен в `.gitpublic/allow`:

```
github.com
get.docker.com
example.com
```

`allow` не заменяет приватные домены. Для замены используй `.gitpublic/replace`:

```
private.company.local ==> example.com
regex:.*\.corp\.internal ==> example.com
```

Аналогия: `scan` говорит: “запретить всё, что совпало с этим шаблоном”. `allow` говорит: “но эти конкретные домены — исключения, их пропустить”.

Коротко:

| Что нужно | Файл |
|---|---|
| удалить файлы | `.gitpublic/ignore` |
| переписать приватный текст/домен/IP | `.gitpublic/replace` |
| упасть, если секрет/домен/IP выжил | `.gitpublic/scan` |
| сделать исключения для публичных доменов, найденных проверкой | `.gitpublic/allow` |

Больше примеров: [расширенная настройка](./docs/ADVANCED.md) / [RU](./docs/ADVANCED.ru.md).

В Git нет «приватного файла в публичном репо». Поэтому нужно два репо. Эта
тулза держит их в синке — без утечек.

### Диагностика установки

```bash
git-private2public doctor
```

`doctor` проверяет Git, `git-filter-repo`, репозиторий и конфигурацию, локальные
файлы правил, pre-push hook и доступ к source/target. Найденные секреты всегда
редактируются: выводятся только тип, расположение, безопасная подсказка из префикса/последних 4 символов, длина и короткий SHA-256,
само значение учётных данных в отчёт не попадает.

### Безопасная транзакция публикации

`publish` запоминает SHA целевой ветки до очистки и использует точный
`--force-with-lease`. Если во время обработки кто-то обновил публичную ветку,
публикация остановится и не затрёт чужие изменения. После push проверяется SHA
на сервере. Репозитории меньше 100 МиБ автоматически клонируются и сканируются
повторно; для больших выполняется проверка SHA без дополнительного клона.
Существующие теги никогда не перезаписываются принудительно.

Для CI команды `scan`, `publish --scan` и `guard run` поддерживают
`--format json`.

Управляемый pre-push dispatcher умеет одновременно запускать guard и
автопубликацию, сохраняя и вызывая уже существующий пользовательский hook.
