**[English](./README.md)** · **[Русский](./README.ru.md)**

**Как `.gitignore`, только для того, что можно публиковать.**

Установи, запусти внутри Git-репозитория и следуй подсказкам.

```bash
pip install git-private2public
git-private2public
```

Этого достаточно, чтобы проверить отслеживаемые файлы и всю Git-историю на известные учётные данные. Конфиг не нужен. В интерактивном терминале программа также предложит включить автоматическую проверку перед каждым `git push`.

## Что ты хочешь сделать?

### Проверить этот репозиторий

```bash
git-private2public
```

Программа сама проверит репозиторий и объяснит, что нашла. Полное значение секрета никогда не выводится.

### Опубликовать безопасную публичную копию

```bash
git-private2public init
git-private2public publish
```

`init` проведёт через одноразовую настройку. `publish` создаст очищенное публичное зеркало, не изменяя приватный репозиторий.

### Что-то не работает

```bash
git-private2public doctor
```

`doctor` проверит Git, конфигурацию, hooks и доступ к обоим репозиториям, а затем скажет, что исправить.

## Просто по умолчанию

- для сканирования ничего настраивать не нужно;
- безопасные правила уже встроены;
- одна понятная команда делает одно понятное действие;
- интерактивный режим сам объясняет следующий шаг;
- сложные настройки доступны только тем, кому они нужны.

Большинству пользователей достаточно:

```text
git-private2public           проверить сейчас
git-private2public init      настроить private → public
git-private2public publish   опубликовать чистую копию
git-private2public doctor    найти проблему
```

Всё остальное — свои правила, замены, JSON для CI, hooks, ветки, теги, YAML и технические детали — вынесено отдельно:

## Безопасность

Публикация выполняется во временном clone. Приватный репозиторий не переписывается. Конкурентные изменения публичной ветки защищены через `--force-with-lease`, отправленный commit проверяется, а существующие теги никогда не перезаписываются принудительно.
